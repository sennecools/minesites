In HeroRender, read `section.settings.backgroundColor` as the section-level background override and apply it to the outer wrapper element of the section.

The override logic:

1. Read: `const sectionBgOverride = section.settings.backgroundColor as string | undefined;`
2. If `sectionBgOverride` is set (non-empty string), use it as the background color for the section's outermost wrapper div
3. If not set (undefined or empty string), fall back to the existing hero background behavior (hero.backgroundType: solid/gradient/image)

The override applies at the SECTION level (outermost wrapper), not inside the hero content area. The existing hero background (solid/gradient/image) can remain for the inner content area, but the outer section wrapper gets the override.

Implementation approach:

- Find the outermost `<div>` or `<section>` wrapper that wraps the entire hero section
- Add a conditional `backgroundColor` to its inline style:
    ```tsx
    style={{
      ...(sectionBgOverride ? { backgroundColor: sectionBgOverride } : {}),
      // ... existing styles if any
    }}
    ```
- If the outermost wrapper currently uses `getBackgroundStyle()` (the function that computes the hero background CSS), keep that behavior but let `sectionBgOverride` take precedence:
    ```tsx
    style={{
      ...(sectionBgOverride ? { backgroundColor: sectionBgOverride } : getBackgroundStyle()),
    }}
    ```

This is a minimal, non-breaking change. The existing `getBackgroundStyle()` function handles solid/gradient/image as before when no override is set.

DO NOT change any of the hero content rendering, typography, CTA buttons, or other logic. Only the background style application on the outermost wrapper changes.

## STRIDE Threat Register

| Threat ID | Category  | Component                                                     | Disposition | Mitigation Plan                                                                                                                                                                                                                                                                                                                                                 |
| --------- | --------- | ------------------------------------------------------------- | ----------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| T-02-10   | Injection | section.settings.backgroundColor rendered as CSS inline style | accept      | React's style prop serializes values through its DOM serializer, which HTML-escapes attribute values. A backgroundColor value of `</style><script>alert(1)</script>` is treated as a CSS color string (which would simply produce an invalid color — no parsing as HTML). The value cannot escape the style attribute boundary when set via React's style prop. |
| T-02-11   | Tampering | section.settings.backgroundColor stored in DB as Json         | accept      | The PUT /api/servers/[serverId] handler already validates ownership. The backgroundColor is stored as part of the section settings JSON — no schema enforcement on the value itself, but since it is only rendered as a CSS property value (not innerHTML or href), the injection surface is minimal and mitigated by React's escaping.                         |
