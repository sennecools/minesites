## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Tampering | Website.subdomain — two users claiming same subdomain | mitigate | `subdomain @unique` constraint in schema enforces at DB level; application-level `findUnique` pre-check in createServer action is defense-in-depth. Both are preserved during the rename (the @unique annotation is on the new Website model). |
| T-06-02 | Tampering | Section.websiteId — client could fabricate a websiteId in PUT body | accept | Ownership check (`existingWebsite.userId !== session.user.id`) is preserved verbatim in the API route rename; no new bypass introduced by Phase 6 renames |
| T-06-03 | Information Disclosure | Prisma migration SQL logged to stdout | accept | Migration runs in dev environment only; no production data present per D-08 (fresh database) |
| T-06-04 | Spoofing | Session/auth changes | accept | No auth changes in Phase 6; NextAuth session is unchanged; User.websites relation is schema-only |
| T-06-05 | Elevation of Privilege | Shadow database creation permission on cloud Postgres | accept | Shadow DB only used by migrate dev, not at runtime; worst case is P3014 error (migrate fails, fallback to db push); no privilege escalation possible |
